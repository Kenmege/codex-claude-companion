Identity checkpoint

The direction is the product.

OpenAI's openai/codex-plugin-cc lets Claude Code invoke Codex. This bridge runs in the opposite direction: Codex supervises Claude through a durable control plane. It is an independent community project by Kennedy Umege and is not affiliated with or endorsed by OpenAI or Anthropic.

↗

Separate system, separate purpose. This bridge is separate from CAMPUS. CAMPUS is not required to install, launch, supervise, recover, or verify a bridge job.

System map

One mission. Four durable hand-offs.

Codex-Claude Bridge adds a durable orchestration layer to the Codex-to-Claude workflow. Initiation happens through the current source build's codex-claude command surface. The repository and npm package are temporarily named codex-plugin-cc during the compatibility-safe migration; that legacy identifier is not the product name.

Initiation sequence

Launch the bridge in five deliberate steps.

Run these steps from the machine hosting Codex, Claude Code and tmux. Replace the example test command and acceptance criteria with the target repository's real gates.

1

Install the current bridge from source

As of 19 July 2026, public npm version 1.1.1 predates the durable bridge. Use a source checkout containing delegate and bridge-doctor, register it with Codex, then restart Codex CLI once.

Terminal - one-time setup
cd /absolute/path/to/codex-plugin-cc
npm install -g .
codex-claude enable
# Restart Codex CLI after enable completes.

Confirm the installed build exposes the new surface with codex-claude --help. The source directory is still named codex-plugin-cc during the staged migration. Do not treat OpenAI's repository of the same name as this bridge, and do not install the proposed scoped package until an approved release publishes it.

2

Prove the control plane is ready

Run both diagnostics. The first checks the full installation and registration path; the second checks the durable bridge primitives and executable compatibility.

Terminal - readiness gate
codex-claude doctor
codex-claude bridge-doctor --json

# Optional: include a live non-interactive Claude probe.
codex-claude doctor --probe-runtime

Continue only when the required checks report ready. A green doctor proves local readiness; the actual mission still has to complete, verify and deliver.

3

Enter the target workspace

Start from the repository to be changed. Preserve its existing worktree state and choose a repository-native verification command before dispatch.

Terminal - mission context
cd /absolute/path/to/project
git status --short

# Confirm Codex supplied the origin thread ID.
printf '%s\n' "$CODEX_THREAD_ID"

If CODEX_THREAD_ID is not present, pass the originating Codex task ID explicitly with --thread-id.

4

Dispatch the implementation mission

This write-capable template gives the worker the current repository, explicit acceptance criteria, one bounded repair attempt and a real independent verification command. Remove --wait when you want the job ID returned immediately for live supervision.

Terminal - durable delegation
codex-claude delegate \
  --thread-id "$CODEX_THREAD_ID" \
  --profile trusted-autonomous \
  --path "$PWD" \
  --repo-root "$PWD" \
  --permitted-root "$PWD" \
  --model opus \
  --effort xhigh \
  --accept "The requested behavior is implemented" \
  --accept "Relevant tests pass with evidence" \
  --verify-command '["npm","test"]' \
  --max-repairs 1 \
  --wait \
  -- "Implement the requested change, preserve existing work, run the relevant tests, and report exact evidence."

Important: --verify-command is a JSON argument array, not a shell string. Use the target project's true test entry point, for example ["go","test","./..."] or ["pytest","-q"].

5

Record the job ID and supervise

For a background dispatch, copy the returned ccb_... ID into the commands below. Follow logs, steer the same live session, and then wait for the verified delivery result.

Terminal - control loop
codex-claude status <ccb_job-id> --json
codex-claude logs <ccb_job-id> --follow --timeout 1800
codex-claude send <ccb_job-id> --wait --message "Focus on the failing integration test."
codex-claude wait <ccb_job-id>
codex-claude status <ccb_job-id> --json
Execution profile

Choose the lane that matches the mission.

The profile controls Claude's interactive permission behavior. It does not replace the task's repository scope, acceptance criteria or independent checks.

Default

Standard

Omit --profile or pass --profile standard. Best for non-writing headless work or jobs that should retain ordinary permission behavior.

Write-capable

Trusted autonomous

Pass --profile trusted-autonomous to run without interactive Claude permission prompts on the same user account. Use it only for a workspace you deliberately trust; it is host trust, not OS isolation.

Live operations

Observe, steer, recover - without losing the job.

The bridge is valuable because execution survives outside the initiating Codex turn. The durable ledger and tmux session remain addressable through the job ID.

See every job

Inspect active and terminal durable jobs.

codex-claude list --json

Inspect one state

Read its worker, verification and delivery state.

codex-claude status <ccb_job-id> --json

Follow execution

Stream the worker log until terminal or timeout.

codex-claude logs <ccb_job-id> --follow --timeout 1800

Attach to tmux

Print the attach command, or attach directly with --exec.

codex-claude attach <ccb_job-id> --exec

Steer the same session

Queue guidance and wait for its replay acknowledgement.

codex-claude send <ccb_job-id> --wait --message "..."

Recover supervision

Resume the broker for one unsettled job or all of them.

codex-claude recover <ccb_job-id>

Answer a question

Correlate the reply when the status contains a pending question ID.

codex-claude send <id> --question-id <q-id> --wait --message "..."

Cancel deliberately

Record a human-readable cancellation reason in the durable lifecycle.

codex-claude cancel <ccb_job-id> --reason "Scope changed"
↗

Live steering has a clear window. Use send while the authoritative Claude worker is running. Once the first authoritative result closes the input window, start a new delegated job for follow-up work. recover restores supervision only for an unsettled job.

Completion contract

Process exit is not the finish line.

A bridge mission is successful only after the worker completes, the origin's independent verification passes, and the result is delivered and acknowledged.

Wait exitMeaningOperator action
0 - CompleteWorker completed, verification passed, and origin delivery was acknowledged.Review the receipt and changed files; the mission can close.
3 - Terminal non-successThe job failed, or verification/delivery did not satisfy the completion gate.Inspect status --json and logs; dispatch a bounded repair if appropriate.
4 - Question pendingClaude needs operator input before it can continue.Read the pending question and reply with send --question-id ... --wait.
Other nonzeroAn operational error occurred, such as a wait timeout or invalid input.Use the diagnostic output and bridge-doctor --json to isolate the control-plane issue.
✓The receipt names the expected workspace and origin.
✓The worker outcome is terminal and explained.
✓Independent verification reports its exact command and evidence.
✓Delivery is acknowledged by the originating Codex task.
✓Changed files match the requested scope.
✓Remaining risks or follow-up work are explicit.
Operator reference

The compact bridge command deck.

Keep this block beside the task. Substitute the job ID and target-specific verification command.

Terminal - complete reference
# One-time source installation and registration (pre-release bridge)
cd /absolute/path/to/codex-plugin-cc
npm install -g .
codex-claude enable

# Readiness
codex-claude doctor
codex-claude bridge-doctor --json

# Enter the target repository; this example returns immediately for background control.
cd /absolute/path/to/project
git status --short

# Dispatch (add --wait for synchronous completion)
codex-claude delegate \
  --thread-id "$CODEX_THREAD_ID" \
  --profile trusted-autonomous \
  --path "$PWD" \
  --repo-root "$PWD" \
  --permitted-root "$PWD" \
  --verify-command '["npm","test"]' \
  --accept "Requested behavior works and tests pass" \
  --max-repairs 1 \
  -- "Implement the requested change and report exact validation evidence."

# Supervise with the returned ccb_... ID
codex-claude list --json
codex-claude status <ccb_job-id> --json
codex-claude logs <ccb_job-id> --follow --timeout 1800
codex-claude send <ccb_job-id> --wait --message "Your guidance"
codex-claude wait <ccb_job-id>
codex-claude status <ccb_job-id> --json

# Recovery and controlled stop
codex-claude recover <ccb_job-id>
codex-claude cancel <ccb_job-id> --reason "Reason"

When launch does not go green.

Plugin not registeredRun codex-claude enable --json.Restart Codex CLI and re-run doctor.

Ready state

Define. Dispatch. Direct. Prove.
